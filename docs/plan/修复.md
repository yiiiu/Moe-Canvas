==================================================
一、先确认阶段二：刷新后 loading 恢复入口提前
==================================================

当前已经做过一个修正：

- AsyncTaskRecord loading 恢复在 main.js 应用早期初始化阶段直接启动。
- 不再依赖任务中心恢复完成。
- 任务中心后续再调用时复用同一个恢复器。

1. 刷新页面后，节点 loading 恢复不再等待任务中心初始化。
2. 节点状态能直接从 AsyncTaskRecord 恢复：
   - isGenerating
   - jobStatus
   - asyncTaskStatus
   - runtimeTaskId
3. 重复安装恢复器不会重复监听。
4. 重复安装恢复器不会重复启动 timer。
5. 如果刷新后仍然需要 5-6 秒才恢复 loading，需要判断是否是：
   - 项目节点本身加载晚
   - 画布节点渲染晚
   - workflow 数据加载晚
   - 节点 DOM 生成晚
   而不是继续补 overlay 延迟。
6. 不要通过加更多 setTimeout 来掩盖问题。
7. 如果恢复入口已提前，但节点还没渲染出来，可以采用“恢复器先启动，等待节点出现后 patch”的方式。
8. loading 恢复必须来自 AsyncTaskRecord 的状态，而不是临时 DOM class。

阶段二通过标准：

- 页面刷新后，AsyncTaskRecord 恢复器在应用早期启动。
- 不依赖任务中心也能恢复节点 loading。
- 重复调用恢复器不会产生重复副作用。
- 不再出现固定等待任务中心导致的 6 秒恢复延迟。
- 如果仍然慢，能通过日志区分是恢复入口慢，还是节点加载/渲染晚。

阶段二确认通过后，再进入下面的阶段三修正。

==================================================
二、阶段三核心问题：无可查询任务 ID 的供应商不能只针对 GRSAI 打补丁
==================================================

当前阶段三遇到的问题：

某些供应商，例如当前 GRSAI，请求后并不会返回一个可用于后续查询任务状态的 taskId / jobId / pollingId。

它可能只返回：

- resultId
- imageId
- recordId
- 最终结果数据
- 图片 URL
- raw result

这导致刷新页面后：

- 没有可用于远端 poll 的 ID
- 节点 loading 无法继续恢复
- 后端可能已经收到结果，但前端无法绑定回原节点
- 如果错误恢复 loading，就会出现无限 loading
- 如果把 resultId 当 taskId，就会进入错误轮询链路

注意：不要只针对 GRSAI 写特判。  
这个问题属于一类通用问题：

“无可查询任务 ID 的供应商如何刷新恢复”。

请把它抽象为 provider capability / recovery capability，而不是写 provider === 'grsai' 的硬编码。

==================================================
三、供应商恢复能力分型
==================================================

请把供应商按恢复能力分型，而不是按供应商名字分支。

至少需要支持以下几类：

1. 可远端轮询型 provider

特征：

- submit 后返回可查询 ID
- 后续可以用该 ID 查询任务状态 / 结果

可能字段：

- taskId
- task_id
- jobId
- job_id
- pollingTaskId
- recoveryTaskId
- queryId

恢复策略：

- recoveryMode = remote_poll
- 需要 queryableTaskId
- 刷新后可以继续远端轮询

2. 即时结果型 provider

特征：

- submit 后不返回可查询任务 ID
- 直接返回最终结果
- 可能返回 resultId，但 resultId 不一定能查询任务

恢复策略：

- recoveryMode = immediate_result
- 请求返回后必须立即写回节点和 AsyncTaskRecord
- status = success
- 不进入远端 poll
- 刷新后有结果就直接 success

3. 本地代理可恢复型 provider

特征：

- 远端不给可查询 taskId
- 但本地后端代理可以用 runtimeTaskId 保存状态 / 结果
- 刷新后前端可以用 runtimeTaskId 查询本地后端恢复状态

恢复策略：

- recoveryMode = local_proxy_poll
- 不用远端 taskId
- 用 runtimeTaskId 查询本地代理
- 后端代理负责维护 runtimeTaskId -> status/result/error

4. 不可恢复型 provider

特征：

- 没有可查询 taskId
- 没有最终 result
- 本地代理也没有恢复能力
- 请求 pending 时刷新后无法恢复

恢复策略：

- recoveryMode = none
- 刷新后进入 interrupted / unrecoverable / request_lost
- 不恢复 loading
- 不重新提交
- 不假装继续执行

==================================================
四、必须明确几个字段的语义
==================================================

请检查 AsyncTaskRecord 结构，并明确这些字段的语义：

1. runtimeTaskId

含义：

- 本地任务唯一 ID
- 前端生成
- 所有异步任务必须有
- 不依赖供应商返回
- 用于绑定节点、画布、本次请求、本地状态、本地恢复

必须绑定：

- nodeId
- canvasId
- provider
- kind
- status
- createdAt
- updatedAt

2. remoteTaskId

含义：

- 远端返回的任务相关 ID
- 可能是 taskId，也可能只是 resultId
- 不默认可用于 poll
- 只做兼容字段或远端元信息

注意：

- 不要默认 remoteTaskId 就是可查询 ID
- 不要把 resultId 误写为可查询 taskId

3. queryableTaskId

含义：

- 已确认可用于 provider.poll() 的 ID
- 只有这个字段存在时，才允许进入 remote_poll 恢复

判断：

- provider.supportsRemotePoll = true
- queryableTaskId 存在
- queryableTaskId 确认来自可查询任务字段

4. remoteResultId / resultId

含义：

- 远端结果 ID
- 只作为结果元信息
- 不能用于任务状态恢复
- 不能当 queryableTaskId

5. recoveryMode

建议枚举：

- remote_poll
- local_proxy_poll
- immediate_result
- none

6. provider capabilities

建议至少包含：

- supportsRemotePoll
- returnsImmediateResult
- supportsLocalProxyRecovery
- requiresQueryableTaskId

不要在恢复逻辑里散落 provider 名字判断。

==================================================
五、发起请求前必须先创建 AsyncTaskRecord
==================================================

对所有异步任务，不管是文本、图片、视频、音频，都必须遵守：

前端在发起供应商请求前，先创建并持久化 AsyncTaskRecord。

也就是：

1. 用户点击生成
2. 立即生成 runtimeTaskId
3. 创建 AsyncTaskRecord
4. 写入 localStorage / AsyncTaskStore
5. 节点进入 loading
6. 再发起 provider 请求

不能等供应商返回后再创建记录。

原因：

用户可能在请求 pending 阶段刷新页面。

如果等响应回来才创建记录，就会出现：

- 请求已经发出
- 页面刷新导致前端上下文丢失
- 后端可能还在处理
- 前端没有 runtimeTaskId 可以恢复
- 节点无法绑定结果

==================================================
六、无 queryableTaskId 的供应商通用处理
==================================================

对所有无 queryableTaskId 的供应商，统一按以下规则处理。

不要只针对 GRSAI。

1. 如果请求返回最终结果

必须立即：

- 写回 node.data
- 写回 AsyncTaskRecord.result / resultSpec
- status = success
- 停止 loading
- 持久化最新状态
- 不进入 poll loop

2. 如果请求还在 pending，用户刷新页面

刷新后：

- 如果本地代理支持 runtimeTaskId 恢复，则进入 local_proxy_poll
- 如果本地代理不支持恢复，则进入 interrupted / request_lost
- 不要自动重新提交请求
- 不要一直 loading
- 不要假装远端可轮询

3. 如果返回的是 resultId

resultId 只能保存为：

- remoteResultId
- resultId
- metadata.resultId

不能保存为：

- queryableTaskId

除非已经确认该 resultId 可以用于远端查询任务状态。

4. 如果 provider 不支持 poll

即使存在 remoteTaskId，也不能进入 remote poll。

remoteTaskId 在这种情况下只是元信息，不是恢复凭据。

==================================================
七、本地代理恢复 local_proxy_poll
==================================================

对于没有远端可查询任务 ID，但可以通过本地代理恢复的供应商，需要建立统一机制。

前端请求本地代理时必须携带：

- runtimeTaskId
- nodeId
- canvasId
- provider
- kind

后端代理需要做：

- 收到请求后创建本地任务记录
- 使用 runtimeTaskId 作为本地恢复 key
- 向上游供应商发起请求
- 上游返回结果后，把结果保存到 runtimeTaskId 对应记录
- 如果失败，把 error 保存到 runtimeTaskId 对应记录
- 提供本地恢复查询接口，用 runtimeTaskId 查询状态 / 结果

刷新后前端流程：

1. 读取 AsyncTaskRecord
2. 发现 provider 不支持 remote_poll
3. 发现 supportsLocalProxyRecovery = true
4. 使用 runtimeTaskId 查询本地后端恢复接口
5. 如果后端返回 running，恢复 loading
6. 如果后端返回 success，写回节点，停止 loading
7. 如果后端返回 failed，写回 error，停止 loading
8. 如果后端没有该 runtimeTaskId，进入 interrupted / request_lost

注意：

- 不要用 resultId 查远端
- 不要重新提交供应商请求
- 不要重复扣费
- 不要无限 loading

==================================================
八、刷新恢复判定表必须升级
==================================================

刷新恢复不能再靠：

“localStorage 里看到一条缓存就恢复 loading”。

必须基于以下维度判断：

- 是否已有有效 result
- 当前 status 是否终态
- provider 是否支持 remote poll
- 是否存在 queryableTaskId
- provider 是否支持 local proxy recovery
- 是否存在 runtimeTaskId
- 是否超过 expiresAt
- 当前请求是否可能已经丢失

建议判定表如下：

1. 有有效 result

刷新后状态：

- success

行为：

- 不恢复 loading
- 不再 poll
- 直接恢复结果

2. 当前 status 是终态

终态包括：

- success
- failed
- canceled
- cancelled
- interrupted
- expired

刷新后状态：

- 保持终态

行为：

- 不恢复 loading
- 不再 poll

3. status 是 running / polling / processing，provider 支持 remote poll，且有 queryableTaskId

刷新后状态：

- recovering / polling

行为：

- 恢复 loading
- 继续 remote poll

4. status 是 running / polling / processing，provider 支持 remote poll，但没有 queryableTaskId

刷新后状态：

- interrupted / missing_queryable_id

行为：

- 不恢复 loading
- 不重新提交
- 不进入 poll

5. status 是 running / polling / processing，provider 不支持 remote poll，但支持 local proxy recovery，且有 runtimeTaskId

刷新后状态：

- recovering / local_proxy_poll

行为：

- 恢复 loading
- 查询本地代理任务状态
- 由本地代理返回 running / success / failed

6. status 是 running / polling / processing，provider 不支持 remote poll，且已有有效 result

刷新后状态：

- success

行为：

- 不恢复 loading
- 直接恢复结果

7. status 是 running / polling / processing，provider 不支持 remote poll，没有 result，也不支持 local proxy recovery

刷新后状态：

- interrupted / unrecoverable

行为：

- 不恢复 loading
- 不重新提交
- 不假 loading

8. 有 remoteTaskId，但 provider 不支持 remote poll

刷新后状态：

- 根据 result / local proxy recovery 判断

行为：

- remoteTaskId 只能作为元信息
- 不能用于 poll

9. provider 支持 remote poll，但 remoteTaskId 不是实际查询 ID

刷新后状态：

- 根据 queryableTaskId 判断

行为：

- 优先使用 queryableTaskId
- remoteTaskId 只做兼容

10. status 是 submitted，请求未返回，无 result，无 queryableTaskId，且无 local proxy recovery

刷新后状态：

- interrupted / request_lost

行为：

- 不恢复 loading
- 不重新提交
- 提示用户重新生成

11. running / polling 超过 expiresAt

刷新后状态：

- interrupted / expired

行为：

- 不恢复 loading
- 不再 poll
- 避免历史任务无限恢复

==================================================
九、有效 result 判断
==================================================

不要只判断 record.result 是否存在。

必须实现有效结果判断。

建议按 kind 判断：

图片任务有效 result：

- imageUrl 有效
- images 非空
- outputUrl 有效
- result.images 非空
- result.url 有效

视频任务有效 result：

- videoUrl 有效
- outputUrl 有效
- result.videoUrl 有效
- result.url 有效

音频任务有效 result：

- audioUrl 有效
- outputUrl 有效
- result.audioUrl 有效
- result.url 有效

文本任务有效 result：

- text 非空
- content 非空
- result.text 非空
- result.content 非空

以下不算有效结果：

- result = null
- result = {}
- result = { raw: {} }
- result 只有 id 但没有可展示内容
- 只有 remoteResultId 但没有可展示内容

十四、最终一句话原则
==================================================

阶段三的核心原则是：

有有效结果，就 success。
有 queryableTaskId 且 provider 支持 remote poll，才 remote poll。
没有 queryableTaskId 但本地代理可恢复，就 local proxy poll。
没有 queryableTaskId、没有结果、也不能本地恢复，就 interrupted。
不要把 resultId 当 taskId。
不要看到缓存就恢复 loading。
runtimeTaskId 是所有异步任务的本地唯一凭据。
